Machine Learning Theory — Interview Study Guide

A self-contained refresher for the DoorDash MLE interview loop: traditional ML in depth (fundamentals → linear models → trees/boosting → feature engineering → evaluation → drift → forecasting), deep learning essentials (nets, backprop, SGD), and DoorDash-flavored ML system design deep dives.

1ML fundamentals

The learning problem, stated precisely

Supervised learning: given samples \((x_i, y_i) \sim P(X,Y)\) i.i.d., find \(f\) in a hypothesis class \(\mathcal{F}\) minimizing expected loss on unseen data:

$$f^* = \arg\min_{f\in\mathcal{F}} \; \mathbb{E}_{(x,y)\sim P}\,[\,\ell(f(x), y)\,] \qquad\text{approximated by}\qquad \hat f = \arg\min_{f} \frac{1}{n}\sum_{i=1}^n \ell(f(x_i), y_i) + \lambda\,\Omega(f)$$

Everything in ML practice lives in the gap between these two: the empirical average (training loss) vs. the expectation (generalization). \(\Omega(f)\) is a regularizer that penalizes complexity to close that gap.

  • Regression: \(y \in \mathbb{R}\), typical loss squared error — the optimal predictor is the conditional mean \(\mathbb{E}[Y|X=x]\). Absolute error ⇒ conditional median; pinball loss ⇒ conditional quantile (crucial for ETA promises and supply buffers — asymmetric costs need quantiles, not means).
  • Classification: \(y \in \{0,1,\dots\}\), loss log-loss/cross-entropy — the optimal scorer is the conditional probability \(P(Y=1|X=x)\). Predicting well-calibrated probabilities is more useful than predicting labels, because downstream systems threshold them differently per use case.
  • Unsupervised: structure without labels — clustering (k-means, DBSCAN), dimensionality reduction (PCA), density/anomaly detection. Mention-level fluency is enough for this loop.
Loss functions as beliefsA loss function encodes what errors cost the business. Squared error punishes large mistakes quadratically (sensitive to outliers); absolute error is robust; Huber interpolates; pinball encodes asymmetry (being 10 min late on an ETA hurts more than 10 min early). Choosing the loss is a business decision — say this in interviews.
Maximum likelihood — the unifying view

Most classical losses are negative log-likelihoods under a noise model. Assume \(y = f(x) + \varepsilon\):

Noise assumptionNLL ⇒ lossModel family
Gaussian \(\varepsilon \sim \mathcal N(0,\sigma^2)\)Squared errorLinear regression / OLS
LaplaceAbsolute errorMedian regression
Bernoulli \(y\sim \text{Bern}(p(x))\)Log loss (cross-entropy)Logistic regression
Poisson (counts)Poisson deviancePoisson GLM (order counts!)

And regularization is MAP estimation: an L2 penalty is a Gaussian prior on weights; L1 is a Laplace prior. This one slide of understanding lets you answer "why this loss?" and "what does the penalty mean?" from first principles.

Train / validation / test discipline & cross-validation
  • Train fits parameters; validation selects hyperparameters/models; test is touched once for the final unbiased estimate. Repeated peeking at test = information leak into modeling choices.
  • k-fold cross-validation: split into \(k\) folds; train on \(k-1\), validate on 1, rotate; average. Uses data efficiently and gives a variance estimate across folds. Typical \(k=5\).
  • Stratified folds preserve class ratios (imbalanced problems). Grouped folds keep entities (a customer, a store) entirely in one fold to prevent identity leakage.
  • Time series must use temporal splits — train on past, validate on future (rolling/expanding origin). Random shuffling leaks the future and inflates metrics badly (§10).
Interview probe"Your offline metrics were great but the model failed in prod — why?" Top causes to enumerate: leakage (features computed with post-outcome data), train/serve skew (different feature code paths online vs. offline), temporal validation done wrong, distribution shift between training window and live traffic, feedback loops from prior model's decisions shaping the data.

2Bias–variance & generalization

The decomposition (be able to derive the idea)

For squared loss at a point \(x\), over resamplings of the training set \(D\):

$$\mathbb{E}_{D,\varepsilon}\big[(y - \hat f_D(x))^2\big] \;=\; \underbrace{\big(f(x) - \mathbb{E}_D[\hat f_D(x)]\big)^2}_{\text{bias}^2} \;+\; \underbrace{\mathbb{E}_D\big[(\hat f_D(x) - \mathbb{E}_D[\hat f_D(x)])^2\big]}_{\text{variance}} \;+\; \underbrace{\sigma^2}_{\text{irreducible noise}}$$
  • Bias: systematic error from a too-simple hypothesis class — the model is wrong on average. Linear model on curved reality.
  • Variance: sensitivity to the particular training sample — retrain on new data, get a different model. Deep tree memorizing noise.
  • Noise: the floor no model beats — genuine randomness in \(y\) given \(x\) (a Dasher hits every red light).

The knobs: more model capacity ⇒ bias ↓ variance ↑. More data ⇒ variance ↓ (bias unchanged). Regularization, bagging, feature selection ⇒ variance ↓ (bias ↑ slightly). Boosting rounds ⇒ bias ↓ (variance ↑ eventually).

Diagnosis: learning curves & the gap
SymptomDiagnosisTreatments
Train error high, val ≈ trainHigh bias (underfit)Bigger model, more/better features, less regularization, train longer
Train error low, val ≫ trainHigh variance (overfit)More data, regularization, early stopping, simpler model, bagging, feature pruning
Both high, gap smallBias + possibly noisy labels/featuresFeature work usually beats model swaps
Val fine, prod badNot bias/variance — shift or leakage§9 monitoring, check pipelines

Learning curves (error vs. training-set size): converging high curves ⇒ bias-limited (more data won't help — say this, it's a classic probe); large persistent gap ⇒ variance-limited (more data will help).

Modern nuance worth one sentenceHeavily overparameterized models (deep nets) can show double descent — test error can improve again past the interpolation point — so "more parameters ⇒ overfitting" isn't a law. Know it exists; don't lead with it.

3Linear regression

OLS: three ways to see it
$$\hat\beta = \arg\min_\beta \|y - X\beta\|_2^2 = (X^\top X)^{-1} X^\top y \quad(\text{when } X^\top X \text{ invertible})$$
  1. Optimization view: convex quadratic; gradient \(\nabla = -2X^\top(y - X\beta)\); setting to zero gives the normal equations \(X^\top X\beta = X^\top y\).
  2. Geometric view: \(X\hat\beta\) is the orthogonal projection of \(y\) onto the column space of \(X\); residuals ⟂ features. This is why adding a feature correlated with existing ones changes coefficients (multicollinearity).
  3. Probabilistic view: MLE under Gaussian noise (§1). Gauss–Markov: OLS is the best linear unbiased estimator under homoscedastic uncorrelated errors.

Assumptions and what breaks when they fail

  • Linearity in parameters (you can still use nonlinear features). Fails ⇒ bias; fix with feature transforms, splines, or move to trees/GBMs.
  • Independent errors. Fails with time series/groups ⇒ standard errors wrong; use grouped/temporal CV and robust SEs.
  • Homoscedasticity. Fails (variance grows with mean, common in delivery times) ⇒ weighted least squares or model \(\log y\).
  • No perfect multicollinearity. Near-collinearity ⇒ huge, unstable coefficients with flipped signs; predictions can still be fine. Detect with VIF; fix by dropping/combining features or ridge.
Ridge, lasso & elastic net
$$\text{Ridge: } \min \|y - X\beta\|^2 + \lambda\|\beta\|_2^2 \Rightarrow \hat\beta = (X^\top X + \lambda I)^{-1}X^\top y \qquad \text{Lasso: } \min \|y - X\beta\|^2 + \lambda\|\beta\|_1$$
  • Ridge (L2): shrinks coefficients smoothly toward zero, never exactly zero; handles collinearity (adds \(\lambda\) to eigenvalues, always invertible); one strong feature split into two correlated copies gets its weight shared.
  • Lasso (L1): produces exact zeros ⇒ embedded feature selection. Geometric intuition: the L1 ball has corners on the axes, and the loss contours typically touch a corner. Among correlated features it picks one arbitrarily — unstable selection; elastic net (L1+L2) fixes this.
  • Standardize features first (penalties are scale-sensitive); don't penalize the intercept; tune \(\lambda\) by CV.
  • Bias–variance link: \(\lambda\) ↑ ⇒ bias ↑ variance ↓ — regularization is the canonical variance-reduction dial.
Interview one-liner"Why might a coefficient have the 'wrong' sign?" — collinearity, omitted-variable bias/confounding, or Simpson's paradox across mixed segments. Regression coefficients are not causal effects without a causal design; for decisions like incentive sizing you want uplift/causal modeling, not raw regression.

4Logistic regression

Model, loss, gradient — cold
Model
$$P(y=1\mid x) = \sigma(w^\top x) = \frac{1}{1 + e^{-w^\top x}} \qquad \text{log-odds: } \; \log\frac{p}{1-p} = w^\top x$$
Loss (negative log-likelihood over data) — convex
$$\mathcal{L}(w) = -\sum_i \big[ y_i \log p_i + (1-y_i)\log(1 - p_i) \big] + \lambda\|w\|^2$$
Gradient (memorize — it's beautiful)
$$\nabla_w \mathcal{L} = \sum_i (p_i - y_i)\, x_i$$
  • The gradient is "(prediction error) × features" — identical in form to linear regression's; this is the GLM pattern, and it's also exactly the last-layer gradient in a neural net classifier (§11).
  • No closed form; fit by gradient descent or Newton/IRLS. Convex ⇒ global optimum.
  • Coefficient reading: \(e^{w_j}\) is the odds multiplier per unit of \(x_j\). "Each extra km of distance multiplies odds of cancellation by 1.15."
  • Perfect separation ⇒ weights diverge to ∞; regularization fixes it (know this classic gotcha).
  • Multiclass: softmax \(p_k = e^{w_k^\top x}/\sum_c e^{w_c^\top x}\) with cross-entropy — same gradient pattern \((p - \text{onehot}(y))\,x\).
Calibration & thresholds — where logistic earns its keep
  • Logistic regression trained on log-loss is usually well-calibrated out of the box: among cases scored 0.3, ~30% are positive. Boosted trees and deep nets often are not ⇒ post-hoc Platt scaling (fit a logistic on the scores) or isotonic regression on a held-out set.
  • Why calibration matters at DoorDash-like companies: scores feed expected-value decisions — e.g., expected lateness cost \(= p(\text{late}) \times \text{cost}\), fraud expected loss, acceptance probability inside dispatch. Miscalibrated probabilities corrupt every downstream optimization even if ranking (AUC) is perfect.
  • Threshold choice is economics, not ML: choose the operating point from the cost matrix — threshold \(t^* = \frac{c_{FP}}{c_{FP}+c_{FN}}\) for calibrated scores minimizing expected cost. Say "the model outputs probabilities; the business chooses thresholds per use case."
  • Measure calibration with reliability curves and Brier score \(= \frac1n\sum(p_i - y_i)^2\) (§8).
Why logistic still mattersStrong baseline, interpretable, trivially cheap at serve time, well-calibrated, and debuggable. The professional take: "I'd baseline with regularized logistic on good features; it often gets within a few points of GBM and tells us what signal exists — then justify complexity by the gap."

5Tree-based models: bagging & boosting

Decision trees (CART) — the base learner

Greedy recursive partitioning: at each node, choose the feature-threshold split that most reduces impurity; predict the leaf mean (regression) or class distribution (classification).

Split criteria
$$\text{Gini: } \sum_k p_k(1-p_k) \qquad \text{Entropy: } -\sum_k p_k \log p_k \qquad \text{Regression: variance reduction (SSE)}$$
  • Strengths: nonlinearities and interactions for free; no scaling needed; mixed feature types; interpretable when small.
  • Weaknesses: high variance (small data changes ⇒ different tree); axis-aligned splits struggle with smooth diagonal boundaries; poor at extrapolation — a tree predicts constants outside the training range (major caveat for forecasting trends, §10).
  • Control complexity via max depth, min samples per leaf, or cost-complexity pruning.

Trees are rarely deployed alone. Two opposite ways to combine them fix their flaws: bagging attacks variance; boosting attacks bias.

Bagging & Random Forests — variance reduction by averaging

Bagging: train \(B\) deep trees on bootstrap resamples; average predictions (or vote). Averaging \(B\) estimators with variance \(\sigma^2\) and pairwise correlation \(\rho\) gives:

$$\operatorname{Var}(\bar f) = \rho\,\sigma^2 + \frac{1-\rho}{B}\,\sigma^2$$

As \(B\to\infty\), only the correlation term survives — so the game is decorrelating trees. Random Forest adds the trick: at each split, consider only a random subset of \(m\) features (\(\sqrt{p}\) classification, \(p/3\) regression) ⇒ trees differ more ⇒ \(\rho\) drops ⇒ ensemble variance drops. That formula-plus-story is the complete answer to "why do random forests work?"

  • OOB (out-of-bag) error: each tree leaves out ~37% of samples (\(\lim (1-1/n)^n = e^{-1}\)); evaluate each point on trees that didn't see it — free cross-validation.
  • Nearly tuning-free, parallelizable, robust to noise, hard to overfit by adding trees (more trees ⇒ variance ↓ monotonic).
  • Bias stays ≈ single deep tree's bias — RF doesn't fix underfitting.
Gradient boosting — the math, done properly

Idea: build an additive model stagewise, each new tree fitting the error signal of the current ensemble:

Additive model, greedy stagewise fitting
$$F_M(x) = \sum_{m=1}^{M} \eta\, h_m(x), \qquad h_m = \arg\min_h \sum_i \ell\big(y_i,\; F_{m-1}(x_i) + h(x_i)\big)$$
Functional gradient descent: fit \(h_m\) to the negative gradient (pseudo-residuals)
$$r_i^{(m)} = -\left.\frac{\partial\, \ell(y_i, F)}{\partial F}\right|_{F = F_{m-1}(x_i)} \qquad \text{squared loss} \Rightarrow r_i = y_i - F_{m-1}(x_i)\ \text{(plain residuals)}$$

So gradient boosting = gradient descent in function space, where each step is a small tree. Log-loss ⇒ pseudo-residuals are \(y_i - p_i\) (same "error × step" pattern again). The learning rate \(\eta\) (shrinkage, 0.05–0.3) trades more trees for better generalization.

XGBoost specifics (know these — it's on your resume and theirs)

Second-order objective with explicit regularization (per boosting round)
$$\text{Obj} \approx \sum_i \big[ g_i\, h(x_i) + \tfrac12 \, h_i\, h(x_i)^2 \big] + \gamma T + \tfrac12 \lambda \sum_{t=1}^{T} w_t^2$$

\(g_i, h_i\): first/second derivatives of the loss; \(T\): number of leaves; \(w_t\): leaf weights.

Closed-form optimal leaf weight and split gain
$$w_t^* = -\frac{\sum_{i\in t} g_i}{\sum_{i\in t} h_i + \lambda} \qquad \text{Gain} = \tfrac12\!\left[\frac{G_L^2}{H_L+\lambda} + \frac{G_R^2}{H_R+\lambda} - \frac{(G_L+G_R)^2}{H_L+H_R+\lambda}\right] - \gamma$$
  • Second-order (Newton-like) step ⇒ better trees per round than first-order; any twice-differentiable loss plugs in (quantile≈, Poisson, ranking losses).
  • Regularization is inside the split criterion (\(\lambda\) shrinks leaf weights, \(\gamma\) is a per-leaf complexity charge ⇒ built-in pruning).
  • Missing values: each split learns a default direction — missingness becomes signal, no imputation required.
  • Histogram-based approximate splits (like LightGBM) for speed; column/row subsampling for decorrelation; monotonic constraints (force prediction ↑ in distance — great for sanity/regulatory needs).
  • Key knobs: n_estimators + learning_rate (with early stopping), max_depth (3–8), min_child_weight, subsample/colsample_bytree, lambda/alpha. Tune with early stopping on a temporal validation set.
Bagging vs. boosting — the compare-and-contrast they will ask

  • Trees don't extrapolate — for trending targets, model the detrended signal or add trend features (§10).
  • Feature importance caveats: gain-based importance biased toward high-cardinality features and splits correlated features arbitrarily; prefer permutation importance or SHAP, and note SHAP explains the model, not causality.
  • Why GBMs beat deep nets on tabular data (usually): heterogeneous features, irregular interactions, modest data sizes, and trees' inductive bias for sharp thresholds fit tabular reality; DL wins with huge data or unstructured/sequence inputs — a balanced take here reads as senior.

6Feature engineering

The catalog (organized by data type)

Numeric

  • Scaling (standardize/min-max) — required for linear/NN/distance models; irrelevant for trees.
  • Log/Box-Cox for heavy-tailed (order values, distances, wait times); winsorize outliers; binning for interpretable nonlinearity.
  • Ratios and interactions encode domain knowledge: items-per-order, tip-per-mile, subtotal/basket-size.

Categorical

  • One-hot: fine to ~50 levels; explodes beyond.
  • Target encoding (mean of \(y\) per category): powerful for high-cardinality (store ID, zip) but the #1 leakage source — must compute out-of-fold and with smoothing toward the global mean for rare levels: \(\tilde y_c = \frac{n_c \bar y_c + \alpha \bar y}{n_c + \alpha}\).
  • Frequency encoding; hashing trick (fixed dimension, collisions acceptable); learned embeddings when using NNs — store/item embeddings capture similarity ("this new thai place behaves like these others").

Temporal (huge for delivery problems)

  • Cyclic encoding: \(\sin/\cos(2\pi\,\text{hour}/24)\) so 23:00 and 00:00 are close; day-of-week, is-holiday, is-game-day.
  • Lags and rolling windows: store's average prep time over trailing 7/28 days, same hour-of-week last week (§10).
  • "Time since" features: minutes since order placed, since Dasher's last delivery.

Geospatial

  • Haversine vs. road distance; geohash/H3 cell IDs as categoricals; zone-level aggregates (demand density, average parking time); weather joins.

Aggregates / "entity profile" features

  • Historical behavior per store/Dasher/customer: acceptance rate, mean prep-time error, cancellation rate. Compute with a time cutoff (only data before the prediction moment) — the classic leakage trap.
Leakage — the career-limiting bug (interview favorite)

Leakage: information available at training time that won't exist (or isn't known yet) at prediction time. Offline metrics soar; production craters.

  • Target leakage: features computed after the outcome (e.g., "actual delivery duration" fields, refund flags in a fraud model that only get set post-investigation).
  • Temporal leakage: aggregates computed over the whole dataset including the future ("store average rating" including the very order you're predicting).
  • Group leakage: the same customer/store in train and test ⇒ the model memorizes identity; use grouped splits.
  • Train/serve skew: offline features from batch tables, online from a different codepath ⇒ silent distribution differences. Mitigation: a feature store with the same computation for both, plus point-in-time-correct backfills; log the served features and train on the logs.
Detection heuristicsToo-good-to-be-true metrics; one feature dominating importance; a "future-dated" audit of each feature's availability timestamp; and backtesting with strict point-in-time joins. Say: "for every feature, I ask — at the millisecond of prediction, is this value known, and is it computed by the same code online?"

7Model training in practice

Hyperparameter tuning & early stopping
  • Search strategies: grid (wasteful beyond 2–3 params) → random search (provably better coverage of what matters) → Bayesian/TPE (Optuna, Hyperopt) with successive halving/Hyperband for compute efficiency.
  • Early stopping (GBMs and NNs): monitor validation loss, stop after \(k\) rounds without improvement. It's regularization — effectively selects model complexity on the validation set.
  • Budget hierarchy worth reciting: better features > more/cleaner data > hyperparameters > architecture swaps — tuning gives the last few percent, not the first twenty.
  • Reproducibility: fixed seeds, versioned data/features/configs (the "model registry + data lineage" sentence belongs in any system-design answer).
Class imbalance, done right
  • First question: do you need labels or probabilities? For decisions, keep the natural distribution, train on log-loss, and choose the threshold from costs (§4). Rebalancing distorts calibration.
  • Techniques when needed: class weights (scale_pos_weight in XGBoost ≈ neg/pos ratio), undersample majority (fast, loses data), oversample/SMOTE (careful: synthetic points can cross class manifolds; never before a temporal split).
  • Metrics: accuracy is meaningless at 1% positives; use PR-AUC, recall@precision, or cost-weighted expected loss (§8).
  • If you rebalance or subsample negatives, recalibrate: adjust the intercept by \(\log(\pi_{\text{true}}/\pi_{\text{train}})\) odds correction or Platt-scale on an unbalanced holdout.
DoorDash flavorFraud, cancellations, and severe-lateness prediction are all rare-event problems. A strong answer: "train on full data with class weights, evaluate PR-AUC and recall at the precision the ops team can action, calibrate scores, and set thresholds by expected cost — fraud reviews cost \$X, missed fraud costs \$Y."
From notebook to production — the training pipeline
  1. Data: point-in-time-correct feature extraction; train/val/test with temporal boundaries; data validation (schema, ranges, null rates — e.g., Great Expectations).
  2. Training: deterministic pipeline (data snapshot → features → model artifact); track experiments (MLflow/W&B); baseline first (persistence/logistic/last-week's-value).
  3. Pre-deployment evaluation: offline metrics per segment (not just global!), calibration, backtest vs. incumbent, latency/memory profile.
  4. Deployment: shadow mode (score live traffic, no actions) → canary/small A/B → full rollout; keep instant rollback to the previous artifact.
  5. Retraining cadence: scheduled (daily/weekly) or trigger-based on drift alarms (§9); always re-run the same eval gate before promotion.

Segment-level evaluation deserves emphasis: a global MAE improvement can hide a regression in one metro or at dinner peak — the exact story interviewers want when they ask "your metric improved but complaints rose — why?"

8Evaluation — the deep dive

Classification metrics: what each one actually measures
$$\text{Precision} = \frac{TP}{TP+FP} \quad \text{Recall (TPR)} = \frac{TP}{TP+FN} \quad \text{FPR} = \frac{FP}{FP+TN} \quad F_1 = \frac{2PR}{P+R}$$
  • ROC-AUC: area under TPR-vs-FPR across thresholds = probability a random positive scores above a random negative (rank quality). Insensitive to class imbalance and to calibration — a model can have AUC 0.95 and useless probabilities.
  • PR-AUC: precision vs. recall; use for rare positives — with 0.5% fraud, FPR barely moves when false positives explode, but precision collapses. PR-AUC shows it; ROC hides it. The baseline PR-AUC equals the positive rate (know this).
  • Log loss / Brier: proper scoring rules — reward calibrated probabilities, not just ranking. Brier decomposes into calibration + refinement.
  • Calibration: reliability diagram (predicted-prob buckets vs. observed frequency), Expected Calibration Error. Essential whenever scores enter expected-value math (§4).
  • Operating-point metrics: recall @ fixed precision (ops capacity), precision @ k (review queue of k cases/day) — usually the metric the business actually feels.
Interview probe"ROC-AUC improved but the product got worse — how?" Possible answers: imbalance made ROC insensitive to the FP flood (check PR); calibration degraded while ranking improved; gains concentrated in a segment that doesn't matter while a critical segment regressed; or the offline label ≠ the online outcome (proxy mismatch).
Regression & forecast metrics
MetricFormulaCharacter / gotcha
MAE\(\frac1n\sum|e_i|\)Robust, in target units; optimal predictor = median
RMSE\(\sqrt{\frac1n\sum e_i^2}\)Punishes large errors; outlier-sensitive; optimal = mean
MAPE\(\frac{100}n\sum\frac{|e_i|}{|y_i|}\)Undefined at 0; asymmetric — favors under-forecasting; dangerous for low-volume series
wMAPE\(\frac{\sum|e_i|}{\sum|y_i|}\)Volume-weighted; the practical default for demand forecasting
Quantile (pinball)\(\max(\tau e, (\tau{-}1)e)\)Evaluates quantile forecasts; asymmetric by design
R²\(1 - \frac{SSE}{SST}\)Relative to mean baseline; can be negative; inflated by trends in time series
  • Match the metric to the decision: ETA shown to customer → penalize lateness more (quantile at τ≈0.7–0.9, or asymmetric cost); demand forecast for staffing → wMAPE + a quantile for the buffer; pricing model → calibration of expected values.
  • Always report against a naive baseline (last value, seasonal naive, global mean): "12% wMAPE" is meaningless until you say the seasonal-naive is 18%.
  • Evaluate by segment and horizon: per-store, per-daypart, per-horizon curves — averages hide the failures that page you at dinner rush.
Ranking metrics (for search / recs questions)
  • Precision@k / Recall@k: relevance in the top-k slate.
  • NDCG@k: graded relevance with log-position discount \(\sum \frac{2^{rel_i}-1}{\log_2(i+1)}\), normalized by the ideal ordering — the standard for search.
  • MRR: mean reciprocal rank of the first relevant item — "how fast do users hit something good".
  • Offline ranking labels come from clicks/conversions ⇒ position bias (items shown higher get clicked more regardless of relevance): mitigate with randomization ("golden" traffic), inverse-propensity weighting, or counterfactual estimators — dropping this shows real depth.
Offline → online: the evaluation funnel
  1. Offline backtest on held-out (temporally later) data vs. incumbent + naive baseline, segmented. Gate: no critical-segment regression.
  2. Shadow mode: deploy alongside prod, score real traffic, log — measures feature availability, latency, score distributions on live data, agreement with incumbent. Catches train/serve skew before users do.
  3. A/B test: randomize (users, or better for marketplace effects: geo/time-split randomization), power the test in advance, run full weekly cycles, monitor guardrails.
  4. Holdback: keep a small control after launch to measure long-term effect and detect slow degradation.

A/B testing essentials you must be able to discuss

  • Primary metric + guardrails (cancellations, Dasher earnings, latency); pre-registered; power analysis determines duration — peeking inflates false positives (or use sequential tests).
  • Interference/network effects: in a marketplace, treating some orders changes the Dasher pool available to control orders — classic SUTVA violation. Remedy: switchback tests (randomize region×time-window units), cluster-randomize by geo. This is THE marketplace-experimentation answer at DoorDash — their dispatch changes are evaluated with switchbacks.
  • Offline gains ≠ online gains: feedback loops, position bias, and proxy-label mismatch. State that you treat offline evals as a filter, online as the verdict.
DoorDash flavorExample metric ladder for an ETA model: offline = quantile loss + calibration by segment/horizon; shadow = feature coverage, p99 latency, score drift; online switchback = on-time rate, customer CSAT/cancellations, order conversion (ETAs shown at checkout affect ordering!); guardrails = Dasher utilization. Showing you can build this ladder for any problem is the "evals maturity" they're screening for.

9Model drift & monitoring

Taxonomy of drift (with the right vocabulary)
  • Data/covariate drift: \(P(X)\) changes, \(P(Y|X)\) stable — new metro launches, holiday mix, weather regime. Model may still be fine where it has support.
  • Label/prior drift: \(P(Y)\) changes — fraud rate doubles after a fraud ring arrives.
  • Concept drift: \(P(Y|X)\) changes — the same features now mean something else (post-COVID prep times; a new Dasher pay policy changes acceptance behavior). Retraining on recent data is the only fix.
  • Upstream/schema drift: a pipeline bug, unit change, or a feature silently going null — in practice the most common "drift" is a broken join, not a changed world. Monitor feature health (null rates, ranges) first.
  • Feedback loops: the model's own actions shape future training data (only offers you make generate acceptance labels). Mitigate with exploration/holdbacks and by logging the action taken.
Detection: statistics & dashboards
PSI (Population Stability Index) — industry standard per feature/score
$$\text{PSI} = \sum_{b} (a_b - e_b)\,\ln\frac{a_b}{e_b} \qquad a_b, e_b: \text{actual/expected share in bucket } b$$

Rules of thumb: < 0.1 stable · 0.1–0.25 investigate · > 0.25 significant shift.

  • Alternatives: KS test (continuous), chi-square (categorical), Jensen–Shannon divergence (symmetric, bounded). At big-data scale, statistical tests trigger on trivial differences — use effect-size thresholds (like PSI) not p-values. Saying that is a senior-signal.
  • Monitor in layers: (1) input feature distributions + health; (2) score distribution (mean, quantiles) — shifts here predict trouble before labels arrive; (3) delayed performance once labels mature (deliveries: minutes; fraud: weeks — design metrics around label latency); (4) business KPIs and guardrails.
  • Slice everything by segment (region, daypart, platform); global stability can mask offsetting segment drifts.

Response playbook

  1. Triage: pipeline bug vs. real-world shift (check feature health first!).
  2. Retrain on a recent window (sliding) or reweight recent data; consider dropping poisoned periods (outages, promos).
  3. Evaluate through the same gate (§7), ship via shadow → canary.
  4. If drift is fast/recurring: shorten retrain cadence, or move volatile signal into features the model reads at serve time (e.g., real-time load features) instead of baking it into weights.

10Forecasting — classical models & XGBoost

Time-series anatomy & the baselines that must be beaten

Decompose: \(y_t = \text{trend} + \text{seasonality} + \text{holidays/events} + \text{remainder}\) (additive; multiplicative when amplitude scales with level — take logs). Food delivery demand has strong multiple seasonality: hour-of-day (lunch/dinner peaks), day-of-week (weekend), plus holidays, weather, and sports events.

  • Stationarity: constant mean/variance/autocovariance over time. Classical models need it — difference away trend (\(\Delta y_t\)), seasonal-difference (\(\Delta_{168}\) for hourly-weekly), log for variance. Test: ADF (unit root).
  • ACF/PACF: autocorrelation diagnostics — spikes at lag 168 (hourly data) reveal weekly seasonality; used to pick ARIMA orders.
  • Naive baselines (report these always): persistence \(\hat y_{t+1} = y_t\); seasonal naive \(\hat y_{t+h} = y_{t+h-168}\) ("same hour last week") — shockingly hard to beat for delivery demand; that's the bar every fancy model must clear.
Classical models: ETS & ARIMA (working fluency)

Exponential smoothing (ETS)

Simple: level only
$$\hat y_{t+1} = \alpha y_t + (1-\alpha)\hat y_t \quad \text{(EWMA; } \alpha \text{ = how fast you forget)}$$
Holt-Winters: level + trend + seasonality (the useful one)
$$\ell_t = \alpha \frac{y_t}{s_{t-m}} + (1-\alpha)(\ell_{t-1} + b_{t-1}),\quad b_t = \beta(\ell_t - \ell_{t-1}) + (1-\beta) b_{t-1},\quad s_t = \gamma \frac{y_t}{\ell_t} + (1-\gamma)s_{t-m}$$

Robust, cheap, interpretable — excellent per-store baseline at scale (fit thousands independently).

ARIMA(p, d, q) × seasonal (P, D, Q)ₘ

$$\text{AR}(p):\; y_t = \sum_{i=1}^{p}\phi_i y_{t-i} + \varepsilon_t \qquad \text{MA}(q):\; y_t = \varepsilon_t + \sum_{j=1}^{q}\theta_j \varepsilon_{t-j} \qquad d:\ \text{differencing order}$$

AR = momentum from past values; MA = correction from past shocks; I = differencing to stationarity. Fit by MLE; select orders via ACF/PACF or AIC (auto-ARIMA). Add external regressors (weather, promos) ⇒ ARIMAX. Strengths: principled prediction intervals, small-data efficiency. Limits: per-series fitting, linear dynamics, awkward with multiple seasonalities (then: STL decomposition + ARIMA on remainder, or Prophet-style curve fitting — worth naming both).

Forecasting with XGBoost (the must-know recipe)

Reframe forecasting as supervised regression on a feature table — one global model across all series (stores × zones), which is the industrial pattern:

Feature recipe for target \(y_{s,t+h}\) (series s, horizon h)

Lags: \(y_{s,t}, y_{s,t-1}, \ldots, y_{s,t-168}\) (incl. same-hour-last-week) · Rolling stats: mean/median/std/min/max over 3h/24h/7d windows · Calendar: hour (sin/cos), day-of-week, holidays, events · Exogenous: weather forecast, promos, new-store flags · Entity: store/zone ID (target-encoded or categorical), store age, cuisine.

  • Multi-horizon strategies: (a) recursive — one one-step model fed back; error compounds; (b) direct — separate model per horizon (or horizon-as-feature); more robust, standard choice; (c) seq2seq NNs when horizons are long and data huge.
  • Validation: rolling-origin backtests only (train ≤ t, test t+1…t+h; slide forward). Never random K-fold. Beware rolling features that peek across the split boundary.
  • Trend caveat: trees can't extrapolate beyond seen values (§5). For growing markets: predict growth/differences, normalize by a trailing average, or hybrid — linear/ETS captures trend, XGBoost models the (stationary) residual. Interviewers love this exact point.
  • Probabilistic output: quantile regression (pinball loss) at τ = 0.1/0.5/0.9 — supply planning needs the distribution, not the mean (§ newsvendor logic in staffing: staff to the fractile that balances under- vs. over-supply cost).
  • Global-vs-local: one global model shares strength across sparse series (new stores!) and beats per-series ARIMA when you have many related series + rich covariates; per-series classical wins for few, long, clean series. State the tradeoff, don't pick dogmatically.
DoorDash flavorCanonical prompt: "Forecast orders per zone per 30-min bucket, 1–24h ahead, to plan Dasher incentives." Strong answer skeleton: global XGBoost, direct multi-horizon, features as above; wMAPE + pinball vs. seasonal-naive baseline, rolling-origin backtest by zone/daypart; quantiles feed the staffing optimizer; monitor with PSI + horizon-sliced wMAPE; retrain daily; cold-start zones via similar-zone pooling. (Note the forecast is consumed by an optimization — tie back to your OR guide's newsvendor/covering models.)

11Neural networks & backpropagation

The multilayer perceptron
Layer-by-layer computation
$$h^{(l)} = \phi\big(W^{(l)} h^{(l-1)} + b^{(l)}\big), \qquad h^{(0)} = x, \qquad \hat y = \text{output}(h^{(L)})$$
  • Why nonlinear activations: stacking linear layers is still linear — \(\phi\) is what buys expressiveness. Universal approximation: one wide hidden layer can approximate any continuous function (existence, not learnability — a good nuance).
  • Activations: ReLU \(\max(0,z)\) (default: cheap, no saturation for \(z>0\); "dying ReLU" if stuck negative — LeakyReLU/GELU fix); sigmoid/tanh saturate ⇒ vanishing gradients in deep stacks (keep sigmoid for output probabilities, not hidden layers).
  • Output + loss pairs: regression → linear + MSE/Huber; binary → sigmoid + log-loss; multiclass → softmax + cross-entropy; quantiles → multiple heads with pinball loss (ETA nets do this).
  • A logistic regression is an MLP with zero hidden layers — the gradients you memorized in §4 are the last layer of every classifier net.
Backprop — derive it once, own it forever

Backprop = the chain rule, organized to reuse computation. Define \(\delta^{(l)} = \partial \mathcal{L} / \partial z^{(l)}\) (gradient w.r.t. pre-activations \(z^{(l)} = W^{(l)}h^{(l-1)} + b^{(l)}\)):

Output layer (softmax or sigmoid + cross-entropy)
$$\delta^{(L)} = \hat y - y \quad \text{(the beautiful cancellation)}$$
Backward recursion
$$\delta^{(l)} = \big(W^{(l+1)\top}\, \delta^{(l+1)}\big) \odot \phi'\big(z^{(l)}\big)$$
Parameter gradients
$$\frac{\partial \mathcal{L}}{\partial W^{(l)}} = \delta^{(l)}\, h^{(l-1)\top} \qquad \frac{\partial \mathcal{L}}{\partial b^{(l)}} = \delta^{(l)}$$
  • Forward pass caches every \(h^{(l)}, z^{(l)}\); backward pass runs the recursion once — total cost ≈ 2–3× a forward pass, independent of parameter count per output. That efficiency is why deep learning works at all.
  • Vanishing/exploding gradients: \(\delta^{(l)}\) is a product of \(W^\top\)s and \(\phi'\)s — repeated multiplication shrinks (sigmoid \(\phi' \le 0.25\)) or blows up. Fixes: ReLU, careful init (He/Xavier — variance-preserving), batch/layer norm, residual connections (gradient highways), gradient clipping.
  • Modern framing: autodiff on a computation graph — you define the forward; PyTorch builds the backward. Backprop is just reverse-mode autodiff.
Architectures in one paragraph each (breadth check)
  • Embeddings: learned dense vectors for categorical entities (stores, items, Dashers) — the bridge from tabular/sparse to neural; power similarity, cold-start-ish generalization, and two-tower retrieval (§13).
  • CNNs: convolution = local weight sharing; translation equivariance; for images and local-pattern sequences.
  • RNN/LSTM: sequential state; LSTM gates fight vanishing gradients; largely superseded by transformers but fine for modest sequence problems.
  • Transformers: self-attention \(\text{softmax}(QK^\top/\sqrt{d})V\) — every position attends to every other; parallelizable; scales; the backbone of LLMs and modern sequence/ETA models. Know the QKV sentence and that positional encodings inject order.
  • Two-tower / DLRM-style models: the workhorses of industrial search/recs — user tower and item tower produce embeddings; dot product for retrieval; deeper interaction models for ranking.
  • When DL over GBM on tabular: very large data, heavy categorical embedding needs, sequence/text/image inputs, multi-task learning, or shared representation across many predictions (multi-task ETA models are DoorDash's public example).

12SGD, optimizers & regularization

Stochastic gradient descent — why noisy is good
$$w_{t+1} = w_t - \eta\, \nabla_w \hat{\mathcal{L}}_{B_t}(w_t), \qquad \hat{\mathcal{L}}_{B_t} = \frac{1}{|B|}\sum_{i \in B_t} \ell_i \quad\text{(minibatch estimate)}$$
  • Why stochastic: full-batch gradients cost \(O(n)\) per step; minibatch gradients are unbiased estimates at \(O(|B|)\) — far more progress per FLOP. The noise also helps escape saddle points/sharp minima (implicit regularization toward flatter, better-generalizing solutions).
  • Batch size: small ⇒ noisy, regularizing, slow wall-clock; large ⇒ stable, hardware-efficient, may generalize slightly worse; learning rate typically scales with batch size.
  • Learning rate is THE hyperparameter: too high diverges, too low crawls. Schedules: warmup (stabilize early steps) then decay (cosine/step). Convergence for convex problems needs \(\sum \eta_t = \infty,\ \sum \eta_t^2 \lt \infty\) — nice theory nugget.

Momentum, RMSProp, Adam

$$\text{Momentum: } v_{t} = \beta v_{t-1} + \nabla \mathcal L,\ \ w \mathrel{-}= \eta v_t \qquad \text{Adam: } w \mathrel{-}= \eta\, \frac{\hat m_t}{\sqrt{\hat v_t} + \epsilon}\ \ \text{(1st/2nd moment EMAs, bias-corrected)}$$
  • Momentum accumulates a velocity — damps oscillation across ravines, accelerates along consistent directions.
  • Adam = momentum + per-parameter adaptive step (divide by RMS of recent gradients) — robust default (\(\eta=10^{-3}\)); AdamW decouples weight decay from the gradient (the correct L2 for Adam).
  • Sane default answer: "AdamW + warmup + cosine decay + gradient clipping; SGD+momentum can generalize marginally better for vision but needs more tuning."
Regularization toolbox for nets
  • Weight decay (L2): same MAP story as ridge (§1).
  • Dropout: randomly zero activations (p≈0.1–0.5) during training ⇒ prevents co-adaptation, approximates ensembling; scale at train time (inverted dropout), disable at inference.
  • Early stopping: validation-loss patience — cheap and effective; it's the NN analog of choosing boosting rounds.
  • Batch norm / layer norm: normalize activations ⇒ smoother optimization, higher usable learning rates, mild regularization (batch noise). LayerNorm for transformers/variable batches.
  • Data augmentation & multi-task learning: more effective than any penalty when applicable; auxiliary tasks act as regularizers (multi-task ETA prediction shares representation across pickup/dropoff/total-time heads).
  • Label smoothing, mixup — name-level awareness suffices.

13ML system design — DoorDash deep dives

The 8-step answer template (mirrors your OR playbook)
  1. Clarify product goal & constraints: who consumes the prediction, latency budget, scale, cost of each error type.
  2. Define the ML problem: unit of prediction, target/label (and its latency!), point estimate vs. distribution.
  3. Data & labels: sources, label quality/bias, feedback loops, cold start.
  4. Features: by entity (store/Dasher/order/zone/time); real-time vs. batch; feature store for train/serve parity.
  5. Model: baseline → GBM → (if justified) NN; calibration; multi-task where labels are related.
  6. Evals: offline metrics + segments + baselines; shadow; switchback/A-B; guardrails. (Lead with this — it's their emphasis.)
  7. Serving: online vs. precomputed, caching, fallbacks (rule-based backup when the model times out), p99 latency.
  8. Monitoring & iteration: drift (PSI), feature health, retrain cadence, incident playbook.
Deep dive 1 — ETA prediction (quoted at checkout & post-order)

Problem shape. Predict delivery time at two moments: checkout quote (affects conversion; no Dasher assigned yet — huge uncertainty) and post-order tracking (updates as events unfold). Total time = prep + wait + Dasher travel legs + handoffs; each component is separately predictable.

  • Target subtlety: quoted ETA is a promise, not a mean — lateness hurts asymmetrically ⇒ predict quantiles (e.g., τ=0.75–0.9) or mean + calibrated distribution, and let product choose the displayed quantile. This single point separates strong from average candidates.
  • Features: store (historical prep-time distribution by daypart, current backlog), order (basket size, item types), marketplace (Dasher availability, current load — real-time features), route (distances, historical travel speeds), temporal (hour, dow, weather).
  • Model evolution story: per-segment gradient boosting on engineered features → multi-task deep model (shared encoder, heads per component/quantile, embeddings for store/zone) — DoorDash publicly moved this way (multi-task, probabilistic ETAs); justify by shared signal + many correlated targets + distribution outputs.
  • Labels arrive in ~an hour ⇒ fast performance monitoring possible; but the displayed ETA changes user behavior (cancellation if late vs. quote) — feedback loop; measure with switchbacks, not just offline error.
  • Evals ladder: offline pinball/MAE by segment & horizon + calibration of the promise quantile → shadow (latency, drift) → switchback on on-time rate, conversion, cancellations; guardrail: Dasher wait at store.
  • Sharp follow-up to expect: "accuracy improved but cancellations rose" — the quote got more honest (longer) in slow conditions, hurting conversion; or on-time improved by inflating quotes (sandbagging) — hence pairing accuracy metrics with quote-length and conversion guardrails.
Deep dive 2 — Demand forecasting → Dasher mobilization

Problem shape. Forecast orders per zone per 30-min bucket, 1–48h out; feed a staffing/incentive optimizer (newsvendor/covering — see the OR guide §11 Ex. 3). This is the cleanest "forecast + optimization" combo interview question, and it plays directly to your strengths.

  • Model: global XGBoost (or LightGBM) with lags/rolling/calendar/weather/promo features, direct multi-horizon, quantile heads (§10); hierarchical consistency (zone forecasts should sum ≈ metro forecast — reconcile top-down or via MinT if pushed).
  • Cold start: new zones/stores — pool via similar-entity features (cuisine, density) in the global model; shrink to priors.
  • Evals: wMAPE + pinball vs. seasonal-naive, sliced by zone size / daypart / horizon; but the real metric is decision quality: cost of under/over-staffing in the downstream optimizer — evaluate the forecast through the decision (backtest: what would staffing have cost with this forecast?). Saying "evaluate the forecast by its decision loss" is a standout moment.
  • Monitoring: event contamination (storms, outages) — flag and exclude from training or add indicators; drift after marketing campaigns.
Deep dive 3 — Store feed ranking / search

Problem shape. Rank stores/items for a hungry user in <100ms. Classic two-stage: retrieval (thousands → hundreds; two-tower embeddings + ANN index + business filters like "actually deliverable in 45 min") then ranking (GBM or deep model on rich cross-features, predicting P(conversion) — possibly multi-task: click, add-to-cart, order, reorder).

  • Labels & bias: clicks/conversions are position-biased and shaped by past rankers ⇒ IPW or randomized "exploration" traffic for unbiased eval; delayed labels (reorder, churn) for long-term value.
  • Objective subtlety: pure P(order) over-shows popular chains ⇒ add diversity, new-store exploration (bandit-style), and marketplace health terms (merchant coverage). Multi-objective = weighted score or constrained re-rank — note the parallel to your OR multi-objective toolbox.
  • Evals: offline NDCG@k/recall@k on IPW-corrected data → A/B on conversion, GMV/session, retention; guardrails: new-merchant impressions, delivery-time distribution of orders placed (fast-ranking can push far stores ⇒ late deliveries!).
  • Serving: precompute store embeddings; cache candidates; feature freshness tiers (real-time availability vs. daily embeddings); fallback = popularity ranker.
Deep dive 4 — Fraud & abuse detection

Problem shape. Score orders/accounts/Dasher-reports for fraud (payment fraud, refund abuse, collusion) in real time; actions range from friction (verification) to block — action severity should scale with score and uncertainty.

  • Rare labels, delayed & biased: chargebacks arrive weeks late; investigated cases are a biased sample (you only label what you flagged) ⇒ maintain a small random-review holdout for unbiased recall estimation — this one sentence is a differentiator.
  • Adversarial drift: fraudsters adapt ⇒ fast retraining, velocity/graph features (shared devices, addresses, payment instruments — entity-linkage graphs), and rules for known patterns alongside ML (rules give interpretability + instant response; ML generalizes — hybrid is the right answer, not either/or).
  • Evals: PR-AUC, recall @ actionable precision, cost-weighted expected loss; segment by fraud type; monitor precision of each action tier; guardrail: false-positive friction on good customers (support tickets, churn).
  • Imbalance handling per §7: class weights + calibrated scores + economic thresholds per action.

Top 10 pitfalls in ML interviews
  1. Random CV on temporal data (leaks future). Temporal/grouped splits always.
  2. Target encoding without out-of-fold computation — leakage.
  3. Optimizing ROC-AUC on a 0.5%-positive problem; ignoring calibration when scores feed decisions.
  4. No naive baseline — every metric needs a "compared to what".
  5. Treating regression coefficients / SHAP values as causal effects.
  6. Proposing deep learning before a GBM baseline on tabular data (or refusing DL where multi-task/embeddings/sequences justify it).
  7. Forgetting label latency and label bias (fraud, ranking) when designing evals.
  8. Ignoring train/serve skew — no feature-store / logged-features story.
  9. A/B testing marketplace changes with user-level randomization — interference; say "switchback".
  10. Forecasting trends with raw tree models — no extrapolation; detrend or hybridize.
